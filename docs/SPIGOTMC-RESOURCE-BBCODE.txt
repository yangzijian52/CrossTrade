[SIZE=5][B]Requirements[/B][/SIZE]
[LIST]
[*]Paper 26.2 only
[*]Java 25 or newer
[*]Vault plus a compatible economy plugin is required for all money transactions
[*]Floodgate is optional and only needed for Bedrock form support
[*]SQLite is bundled in the plugin jar
[*]CrossTrade 2.2.0 was successfully tested on a live Paper 26.2 server; 2.2.1 only adds publishing documents and metadata
[/LIST]

[SIZE=5][B]Installation[/B][/SIZE]
[LIST=1]
[*]Stop the server.
[*]Install Java 25 and Paper 26.2.
[*]Install Vault and an economy plugin if you want purchases, payouts or money-based direct trades.
[*]Install Floodgate only if Bedrock form support is required.
[*]Place CrossTrade-2.2.1.jar in the server plugins directory.
[*]Start the server once.
[*]Edit plugins/CrossTrade/config.yml and plugins/CrossTrade/gui.yml.
[*]Restart the server after major configuration changes.
[/LIST]

[CODE]crosstrade.use                         Default: true

You can add permission nodes such as crosstrade.market.listings.20 to give selected players a higher active-listing limit. CrossTrade uses the highest numeric listings permission it finds.

[SIZE=5][B]Direct Trade Rules[/B][/SIZE]
[LIST]
[*]Players can use /trade <player> or sneak-right-click a player.
[*]The market center can also show online players for remote trade requests.
[*]Requests have a unique ID, cooldown and expiration.
[*]Changing items or money resets both confirmations.
[*]Only one countdown task can exist for a trade session.
[*]The final completion check validates both inventories and the latest Vault balances.
[*]Money is settled by net difference when both sides offer money.
[*]Closing the GUI, leaving, dying, changing worlds or server shutdown safely cancels the trade.
[/LIST]

[SIZE=5][B]Market Listing Flow[/B][/SIZE]
[LIST=1]
[*]Run /market sell.
[*]Select one item type from the inventory mirror.
[*]Enter the market display name.
[*]Choose or enter the quantity.
[*]Enter the unit price.
[*]Choose a listing duration from 1 to 10 days.
[*]Review the final preview.
[*]Confirm the listing.
[/LIST]

CrossTrade removes items only during final confirmation. If the inventory changed or there are not enough matching items anymore, the listing is rejected.

[SIZE=5][B]Buying Flow[/B][/SIZE]
[LIST=1]
[*]Open /market or /market plaza.
[*]Choose a seller.
[*]Choose an active listing.
[*]Select the purchase quantity.
[*]Review unit price, total price, balance and required inventory space.
[*]Confirm the purchase.
[/LIST]

CrossTrade does not automatically reduce the purchase quantity when inventory space is insufficient. The selected quantity must be delivered completely or the purchase fails.

[SIZE=5][B]Mailbox[/B][/SIZE]
The mailbox stores returned and recovered items when they cannot be placed directly into a player's inventory. This includes expired listings, cancelled listings, forced admin removals, direct-trade cancellation leftovers and exceptional recovery items.

[CODE]/market mailbox[/CODE]

Players can claim entries one by one or claim all. Items that still do not fit remain safely stored.

[SIZE=5][B]Seller Earnings[/B][/SIZE]
Seller money is written to a payout ledger. If Vault payment succeeds immediately, the seller's balance rises automatically. If payment fails or cannot be safely completed, the payout remains claimable.

[CODE]/market earnings[/CODE]

Payout states include pending, paying, paid and review. Review means an administrator should inspect the economy state before forcing compensation.

[SIZE=5][B]Item Safety[/B][/SIZE]
[LIST]
[*]Market items are stored with Paper ItemStack byte serialization.
[*]A SHA-256 fingerprint is stored for auditing.
[*]Different durability, enchantments, display names, lore, attributes, PDC, custom model data or item components create different item types.
[*]Shulker boxes with different contents are treated as different items.
[*]Container previews are read-only.
[*]Returned items are saved to the mailbox when inventory space is insufficient.
[/LIST]

[SIZE=5][B]Database and Backups[/B][/SIZE]
The default database path is plugins/CrossTrade/data.db. CrossTrade enables SQLite foreign keys, WAL, busy timeout and schema migrations.

Use the admin backup command before manual database maintenance:

[CODE]/market admin backup[/CODE]

Do not manually edit a running database. Stop the server first if direct maintenance is required.

[SIZE=5][B]Upgrading[/B][/SIZE]
[LIST=1]
[*]Stop the server.
[*]Back up the old jar and plugins/CrossTrade.
[*]Replace the jar with CrossTrade-2.2.1.jar.
[*]Start the server.
[*]Check console output for Vault, Floodgate and SQLite initialization messages.
[*]Run a small test trade, listing, purchase, cancellation and mailbox claim before production use.
[/LIST]

CrossTrade 2.0.x databases are migrated automatically to the current schema. CrossTrade 1.x did not have the player-market database, so there is no old market data to migrate.

[SIZE=5][B]Known Limits[/B][/SIZE]
[LIST]
[*]Paper 26.2 is the supported target. Generic Spigot support is not claimed.
[*]Vault and SQLite cannot provide one distributed atomic transaction. CrossTrade uses journal states, rollback, mailbox recovery and review records to reduce risk.
[*]Different economy plugins handle offline-player deposits differently.
[*]Floodgate forms require Floodgate and its form API at runtime.
[*]Do not use /reload or hot-unload tools.
[*]The SpigotMC resource page and support are English-only. Chinese-language support is not provided on SpigotMC.
[/LIST]
